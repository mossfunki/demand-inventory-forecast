# Demand Forecasting & Inventory Optimization

**Problem:** Static reorder models fail in Seattle because they ignore event-driven demand spikes — Seahawks games, PAX West, and Mariners season create predictable surges that a fixed ROP misses entirely. The result is stockouts during events and excess carrying costs in the quiet weeks between.

**Solution:** A weekly-updating demand forecast (Prophet + event regressors) that feeds directly into dynamic safety stock calculations and reorder points — replacing static tables with forward-looking signals.

---

## Results (8-week held-out test set)

| SKU | Prophet MAPE | XGBoost MAPE | Stockout Reduction |
|---|---|---|---|
| Beverages | 7.8% | 11.2% | 33% |
| Snacks | 8.1% | 12.4% | 28% |
| Merchandise | 9.2% | 13.8% | 41% |
| Accessories | 8.0% | 11.9% | 35% |
| Apparel | 8.6% | 14.1% | 29% |
| **Average** | **8.3%** | **12.7%** | **33%** |

Dynamic ROP reduced simulated stockout events by **33%** on average vs. a static fixed reorder point — at equivalent safety stock investment.

---

## Stack

- **Forecasting:** Facebook Prophet with Seattle event calendar as external regressors
- **Comparison model:** XGBoost GradientBoostingRegressor with engineered event features
- **Baseline decomposition:** STL (Seasonal-Trend-Loess) for trend/seasonality separation
- **Inventory math:** Standard safety stock formula `SS = Z × σ_LT × √LT` at 95% service level; dynamic ROP = `(avg_daily_demand × LT) + SS`
- **Dashboard:** Streamlit + Plotly — designed for an ops manager, not a data scientist

## Data note

All demand data is **simulated** using realistic retail parameters calibrated to BLS retail trade indices. The event calendar is built from real public sources (Seattle Special Events Office, team schedules). WSDOT volume data is used as an event attendance proxy and is available via the [WSDOT open data portal](https://wsdot.wa.gov/data/tools).

---

## Key design decisions

**Why Prophet over ARIMA?**  Prophet handles multiple seasonalities (weekly, annual) and accepts external regressors cleanly — event attendance buckets plug in directly. ARIMA requires separate exogenous variable handling that adds complexity without accuracy gains here.

**Why simulate data?**  Retail demand data is proprietary. Simulating from realistic parameters (BLS indices, known event uplift patterns) keeps the methodology transparent and reproducible — any analyst can inspect the generation logic and substitute real data.

**Why 95% service level?**  Standard for consumer retail. Adjustable in `src/inventory.py` — set `SERVICE_LEVEL = 0.98` for high-value SKUs or `0.90` for commodity items.
